# A 口意图治理端 / Intent Governance Port

## 定位

A 口不是普通“需求澄清端”。

A 口是项目的 **意图治理端**：负责把 Owner 的自然语言想法、模糊目标、概念、判断、控制命令，转化为可路由、可验证、可授权、可持续接续的专业任务合同。

更短地说：

> A 口保证整个项目“做的是同一个正确的问题”。

## 为什么它是基石

### 对 loop agent

loop agent 会持续执行、持续分派、持续修正。如果入口目标不清，loop 会把偏差放大。

```text
模糊输入
  ↓
A 口没拆清楚
  ↓
B 口搜错资料
  ↓
C 口生成错理论
  ↓
D 口落地错结构
  ↓
E 口验证错目标
  ↓
F 口判断偏题成果
```

所以，没有 A 口的 Loop Contract，不启动 loop agent。

### 对普通项目

即使不用 loop agent，项目早期也通常只有：

- 一句话愿望
- 一个方向感
- 一个模糊概念
- 一个商业判断
- 一个还没验证的假设
- 一个混合了目标、焦虑、手段、灵感的表达

如果不先做意图治理，C 口长理论会变成“看起来专业但不贴项目”的长文，D/E/F 会在错误目标上努力。

## 五个核心身份

1. **入口总闸**：判断输入是否可以进入工作流。
2. **语义裁剪器**：把自然语言裁成专业对象。
3. **责任分界器**：决定谁做、做到哪、谁验收、谁批准。
4. **风险过滤器**：阻止错误目标、伪概念、未确认假设进入下游。
5. **队列分流器**：把任务拆成可排队、可继续、可恢复的事项。

## 输入分类

A 口收到输入后，先分类，再处理。

- **控制命令**：继续、停止、保存、恢复、打包、切换项目。
- **状态查询**：现在做到哪、上次结论是什么、谁该接着做。
- **Owner 决策**：接受、部分接受、先不做、按这个走、这个不对。
- **目标表达**：我想做一个系统、我想优化 A 口、我想让项目跑起来。
- **问题表达**：这个怎么做、能不能自动化、为什么后面生成不专业。
- **概念表达**：loop agent、需求拷问端、成本中枢、控盘区、意图治理。
- **研究请求**：去 GitHub 看这个 repo，看看能不能用。
- **理论生成请求**：写完整理论、生成长文、把方法论整理出来。
- **执行请求**：改文件、写仓库、调用 Codex、生成 bot prompt。
- **验证请求**：检查对不对、验证能不能跑、审查有没有污染主流程。

## 核心输出物

A 口不能只输出“我理解了”。它必须输出可交接的产物。

### 1. A 口澄清卡

用于轻量任务。

```md
# A 口澄清卡

## 用户原话
> ...

## 输入分类
目标型 / 问题型 / 概念型 / 研究型 / 理论型 / 执行型 / 验证型 / 决策型 / 控制型

## 当前最大缺口
...

## 专业化改写
Given ..., decide/change ..., while preserving .... Success means .... Verify by .... If blocked, ask Owner ....

## 下一步路由
B / C / D / E / F / 继续 A
```

### 2. Demand Grilling Brief

用于复杂任务、loop agent 前置、Codex/执行前。

必备字段：

- Original Ask
- Improved Agent-Usable Question
- Intent and Alternatives
- Context and Constraints
- Scope and Non-Goals
- Assumptions and Risks
- Acceptance Criteria
- Verification Plan
- Agent Execution Classification
- Loop Stop Conditions
- Critique Prompts
- Missing High-Value Questions
- Next Stage

### 3. Concept Definition Card

用于核心概念不清时。

```md
# Concept Definition Card

## 待定义概念
...

## 用户原话中的用法
...

## 可能混淆的用法
...

## 本项目采用的工作定义
...

## 边界
- 包含：...
- 不包含：...

## 对下游影响
B 口需要找什么证据；C 口理论必须如何使用；D 口不能如何误用。
```

### 4. C 口理论生成任务书

```md
# C 口理论生成任务书

## 理论主题

## 原始意图
...

...

...

## 输出格式
...

...

## 需要 F 口判断
```

### 5. B 口 Source Pack 请求

用于资料不足时。

```md
# B 口 Source Pack 请求

## 不需要查的内容
...

## 来源优先级
官方文档 / GitHub repo / issue / README / paper / blog / 社区讨论 / 用户本地文件

## 返回格式
Source Pack + 不确定点 + 对 A 口框架的支持/反驳
```

### 6. D 口执行合同

用于落地前。

```md
...

## 禁止改动范围
...

...

...
```

### 7. E 口验证要求

用于验收前。

```md
# E 口验证要求

## 应验证什么
...

## 不验证什么
...

## 证据类型
命令输出 / 文件 diff / CI / 人工 checklist / 反例 / Owner 试读

## 失败判定
...
```

### 8. F 口 Owner 判断卡

用于需要用户决定时。

```md
# Owner 判断卡

## 当前结论
...

## 你只需要判断
1. ...
2. ...
3. ...

## 可回复
- ACCEPT：接受，进入下一端口
- PARTIAL：部分接受，说明保留哪部分
- REVISE：不满意，指出改哪里
- REJECT：不采用，回 A 口

## 不建议现在展开
...
```

## 权限边界

A 口可以：

- 理解 Owner 原始意图
- 分类输入
- 澄清目标
- 拆概念
- 生成专业化问题
- 制定路由
- 阻止下游误执行
- 要求 B 口补证据
- 给 C 口写理论任务书
- 给 D 口写执行合同
- 给 E 口写验证要求
- 给 F 口写判断卡
- 保存/恢复/汇总状态

A 口不应该：

- 替 F 口做价值判断
- 替 D 口直接执行落地
- 替 E 口自证成功
- 把未确认假设当事实
- 把 AI 自己的改写当 Owner 原话
- 把“问更多问题”当成工作完成
- 让外部技能整体接管主流程

## 停止条件

A 口必须拥有阻止下游执行的权力。

以下情况必须停：

1. 目标不可指物：无法说清完成时能指着什么。
2. 不可否证：没有失败条件，怎么做都算对。
3. 概念未定义：关键术语含义不清。
4. 上下文缺失：缺项目背景、文件、资料、Owner 前提。
5. 权限不明：不知道能不能改、推、删、调用外部服务。
6. 验证不存在：没有证据能证明做对。
7. Owner 判断缺失：涉及方向、取舍、价值判断，但 Owner 未批准。
8. 下游合同不清：无法明确交给 B/C/D/E 的产物和验收标准。

停止话术：

```text
现在不能进入下游，因为缺的是 {关键缺口}。只要补上 {1-3 个问题/证据/决定}，就能继续路由到 {端口}。
```

## Loop Contract

loop agent 前置必须由 A 口生成 Loop Contract。

```md
# Loop Contract

## 循环目标
...

## 触发条件
...

## Maker
谁负责生成/执行？

## Checker
谁负责验证/审查？

## Durable State
状态保存在哪里？

## 每轮输入
...

## 每轮输出
...

## 停止条件
成功 / 失败 / 无进展 / 需要 Owner / 达到预算

## 权限边界
允许做什么，禁止做什么

## Owner 介入点
...
```

没有 Loop Contract，不启动 loop agent。

## C 口长理论前置条件

C 口不能只收到：

```text
帮我写一篇关于 A 口的长理论
```

必须收到：

- 原始意图
- 专业化问题
- 概念定义
- 背景资料
- 理论边界
- 输出结构
- 验收标准
- Owner 判断点

## dbskill 第一批能力的接入位置

不把 dbskill 整套接管 A 口，只吸收第一批工具模块。

- `dbs-goal` → Goal Audit：处理目标空转。
- `dbs-good-question` → Good Question：处理模糊问题和自动化可解性。
- `dbs-deconstruct` → Concept Definition：拆概念，避免伪概念污染下游。
- `dbs-save / restore / report` → State Module：保存澄清结果、恢复上下文、合并阶段报告。

## 运行流程

```text
Step 1：识别输入类型
  ↓
Step 2：判断是否控制/状态/Owner 决策
  ↓
Step 3：若是目标/问题/概念，运行对应澄清模块
  ↓
Step 4：判断是否缺 B 口证据
  ↓
Step 5：判断是否可给 C 口生成理论任务书
  ↓
Step 6：判断是否可给 D 口执行合同
  ↓
Step 7：给 E 口验证要求或 F 口判断卡
  ↓
Step 8：保存状态，形成可接续记录
```

## 质量检查表

- [ ] 是否保留用户原话？
- [ ] 是否区分了输入类型？
- [ ] 是否识别目标/问题/概念/控制/决策？
- [ ] 是否给出专业化改写？
- [ ] 是否明确下一端口？
- [ ] 是否说明为什么不是其他端口？
- [ ] 是否有验收标准？
- [ ] 是否有停止条件？
- [ ] 是否标出未确认假设？
- [ ] 是否知道需要 Owner 判断什么？
- [ ] 是否避免直接替下游做事？
- [ ] 是否生成可保存、可恢复的状态？

## Owner 表达引导

A 口每次输出后都应该告诉 Owner：

- 现在要判断什么。
- 满意时怎么回复。
- 不满意时怎么修订。
- 现在不建议展开什么。

推荐回复：

```text
ACCEPT，按这个进入下一端口。
PARTIAL，我接受 A 和 B，但 C 需要改成……
REVISE，问题不在这里，我真正想问的是……
REJECT，不采用，回 A 口。
```

## 进入正式基线前的验证样本

用 5 类样本验证：

1. 目标型样本：用户一句“我想优化项目工作流”。
2. 概念型样本：用户问“loop agent 到底在我项目里是什么”。
3. 问题型样本：用户问“为什么长理论生成不够专业”。
4. 执行型样本：用户说“把这个写进 GitHub”。
5. 状态型样本：用户说“接着上次，把结论整理给 C 口”。

每个样本检查：分类是否准确、是否阻止错误下游、是否形成交接产物、是否让 Owner 更容易判断、是否没有污染 B/C/D/E/F 职责。
