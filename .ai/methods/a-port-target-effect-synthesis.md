## 0. 最终想要的效果，一句话

你不是想要一个“更会提问的机器人”。

你想要的是一个 **项目入口治理系统**：

> 当 Owner 用普通自然语言说出一个想法时，A 口能把它转成专业、可验证、可路由、可持续接续的任务合同；再决定该进 B 口资料、C 口理论、D 口落地、E 口验证，还是 F 口判断。这样无论是否使用 loop agent，项目都不会在模糊意图上盲目推进。

更短地说：

```text
自然语言想法 → 专业问题表达 → 资料/理论/执行/验证/决策的正确路由
```

## 1. 今天讨论的核心变化

今天的重点不是“dbskill 里哪些技能有用”，而是认清：

```text
A 口/需求拷问端，是整个项目运行发展的基石。
```

一开始我们是在看 dbskill：

- `goal`
- `good-question`
- `deconstruct`
- `save / restore / report`

后来收束出更高层的判断：

> 这些技能只是 A 口里的工具模块，不是 A 口本身。真正的 A 口要治理整个项目入口。

所以今天完成了一次概念升级：

```text
需求澄清端
  ↓
需求拷问端
  ↓
A 口意图治理端 / Intent Governance Port
```

---

## 2. 你真正想解决的问题

你担心的不是“AI 不会写”。

你担心的是：

1. 项目想法本身没有被深入理解。
2. Owner 的自然语言被 AI 过早改写，丢掉真实意图。
3. 核心概念没拆清楚，后续理论建立在伪概念上。
4. B 口不知道该查什么，资料包变成泛泛搜索。
5. C 口没有足够输入，只能写出看似专业的长文。
6. D 口过早落地，把错误目标写进仓库。
7. E 口只能验证“有没有做”，不能验证“是不是做对”。
8. F 口最后要判断一堆前提不清的成果。

所以你想要的不是单点提示词优化，而是一套入口控制机制。

## 3. A 口应该达到的运行效果

### 3.1 Owner 可以用普通话说想法

Owner 不需要一开始就说得像工程师、产品经理、架构师。

可以这样说：

```text
我觉得这个端很重要，但还没想清楚怎么放进项目。
```

A 口应该能接住，并转成：

```text
这是一个项目入口治理问题：需要定义 A 口职责、边界、输出物、停止条件，以及它如何支撑 B/C/D/E/F。
```

### 3.2 A 口自动识别输入类型

A 口要先判断这句话是什么类型：

- 控制命令
- 状态查询
- Owner 决策
- 目标表达
- 问题表达
- 概念表达
- 研究请求
- 理论生成请求
- 执行请求
- 验证请求

不同类型走不同处理方式。

### 3.3 A 口把自然语言专业化

A 口不是简单复述用户话，而是做专业化转换：

```text
用户原话
  ↓
保留原话
  ↓
识别类型
  ↓
指出最大缺口
  ↓
生成专业化问题
  ↓
定义下一端口和交付物
```

注意：专业化不是替用户脑补，而是在保留原话的基础上把含义裁清楚。

### 3.4 A 口敢于阻止下游

A 口成熟以后，最重要的能力不是“继续推进”，而是“该停就停”。

必须停的情况：

- 目标不可指物
- 没有失败条件
- 概念未定义
- 资料缺失
- 权限不明
- 验证证据不存在
- Owner 决策缺失
- 下游合同不清

这不是拖慢项目，而是防止后面高效做错。

## 4. 你想要的 A-F 效果

### A 口：意图治理

输入：Owner 自然语言、控制命令、状态、概念、目标、问题。

输出：

- A 口澄清卡
- Demand Grilling Brief
- Concept Definition Card
- B 口 Source Pack 请求
- C 口理论生成任务书
- D 口执行合同
- E 口验证要求
- F 口 Owner 判断卡

核心效果：

```text
不让模糊输入污染下游。
```

### B 口：证据与背景压缩

输入：A 口定义好的 Source Pack 请求。

输出：

- Search Strategy Brief
- Source Pack
- 支持/反驳 A 口框架的证据
- 不确定点
- 给 C/D/E/F 的影响

核心效果：

```text
不是乱搜，而是按 A 口定义的问题压缩证据。
```

### C 口：理论与方案生成

输入：A 口任务书 + B 口 Source Pack。

输出：

- 长理论
- 方法框架
- 方案分支
- 关键假设
- D/E/F 交接说明

核心效果：

```text
理论不是泛写，而是有根、有证据、有边界、有验收标准。
```

### D 口：仓库落地 / 执行

输入：A 口执行合同 + C 口方案。

输出：

- 文件改动
- 代码/文档落地
- Codex 执行结果
- Git diff / commit / PR

核心效果：

```text
只执行被授权、被定义、可验证的任务。
```

### E 口：验证 / 审查

输入：A 口原目标 + D 口结果。

输出：

- 验证报告
- 证据
- 失败点
- 是否通过
- 是否需要回 A/C/D

核心效果：

```text
验证“做对没做对”，不是只看“做没做”。
```

### F 口：Owner 决策

输入：A/E 给出的判断卡。

输出：

- ACCEPT
- PARTIAL
- REVISE
- REJECT

核心效果：

```text
Owner 不被迫读一堆散乱信息，只判断关键取舍。
```

## 5. 与 loop agent 的关系

你今天的关键判断是：

```text
loop agent 不是先跑起来再说，而是必须先有 A 口边界。
```

loop agent 如果没有 A 口控制，会变成：

```text
自动化扩大混乱
```

有 A 口控制，才可能变成：

```text
自动化推进清楚的任务
```

所以你要的 loop agent 不是“AI 自己循环干活”，而是：

```text
A 口定义循环合同
B/C/D 执行各自任务
E 口验证每轮结果
F 口在关键点批准
状态持续保存和接续
```

Loop Contract 必须先定义：

- 循环目标
- 触发条件
- Maker
- Checker
- Durable State
- 每轮输入
- 每轮输出
- 停止条件
- 权限边界
- Owner 介入点

没有这个，就不启动 loop。

## 6. 与 C 口长理论的关系

你今天还指出：

> 如果项目想法没有被深入了解，没有多方面理解，没有专业表达，后续长理论不可能真正专业。

这句话是核心。

C 口长理论质量不只取决于模型能力，而取决于输入质量。

C 口必须收到：

- Owner 原始意图
- A 口专业化问题
- 核心概念定义
- B 口资料与证据
- 理论边界
- 必须回答的问题
- 不得做的事
- 输出结构
- 验收标准
- F 口判断点

否则 C 口写出来的是：

```text
没有根的长文
```

你想要的是：

```text
由 A 口定题、B 口供证、C 口成体系、D 口落地、E 口验收、F 口决策的长理论生成链
```

## 7. dbskill 在这里的真实位置

dbskill 不是你的主流程。

它是 A 口里的局部工具库。

当前第一批只采用：

- `goal`：判断目标是否空转
- `good-question`：把模糊问题变成可推理、可验证的问题说明书
- `deconstruct`：拆概念，避免伪概念污染下游
- `save / restore / report`：让状态可保存、可恢复、可汇总

正确关系是：

```text
A 口意图治理端
  ├─ Goal Audit
  ├─ Good Question
  ├─ Concept Definition
  └─ State Module
```

不是：

```text
dbskill 接管 A 口
```

## 8. 你最终想做成的系统形态

最终你想要的是一个 **多端口、可接续、可验证、可授权的项目运行系统**。

它的运行形态是：

```text
Owner 用自然语言提出想法
  ↓
A 口意图治理：分类、澄清、专业化、定边界、定路由
  ↓
B 口资料压缩：按问题找证据，不乱搜
  ↓
C 口理论生成：基于 A+B 生成专业长理论或方案
  ↓
D 口执行落地：只做被授权、被定义的任务
  ↓
E 口验证审查：验证是否做对，给证据
  ↓
F 口 Owner 决策：接受、部分接受、修订、拒绝
  ↓
状态回填：下次可以接着，不从零开始
```

这不是单个提示词，也不是一个聊天机器人。

它是一个把项目从“想法”推进到“可验证成果”的治理循环。

## 9. 成功后的体验应该是什么样

### 对 Owner 来说

Owner 可以这样说：

```text
我觉得这个 GitHub repo 里的技能可能有用，看看怎么放进我的工作流。
```

系统应该自动变成：

```text
A：这属于外部技能采纳问题，先定义采纳标准和端口影响。
B：扫描 repo，返回 Source Pack。
A：判断哪些技能进入 A/B/C/D/E/F。
C：生成接入理论和候选方案。
D：写入仓库候选文档。
E：检查是否污染主流程。
F：Owner 决定 ACCEPT/PARTIAL/REVISE/REJECT。
```

Owner 不需要自己一开始就说出完整工程语言。

### 对项目来说

项目不会再靠聊天记忆推进，而是靠：

- 文件
- PR
- Source Pack
- 任务书
- 执行合同
- 验证报告
- Owner 判断卡
- 状态回填

### 对 AI 来说

AI 不再“想到哪写到哪”，而是被端口合同约束：

- A 定义问题
- B 压缩证据
- C 生成理论
- D 落地执行
- E 验证审查
- F 决策授权
