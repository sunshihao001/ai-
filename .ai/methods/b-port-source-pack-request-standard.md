# B 口 Source Pack 请求标准

## 定位

B 口不是普通搜索端，也不是资料堆砌端。

B 口是 **证据与背景压缩端 / Source Pack Port**。它接收 A 口定义好的研究问题，把外部资料、GitHub 仓库、本地文件、案例、文档、争议点压缩成 C/D/E/F 可用的 Source Pack。

B 口不能从模糊需求直接开始大搜。B 口必须先拿到 A 口的 Source Pack 请求。

---

## A 口给 B 口的输入必须包含

```md
# B 口 Source Pack 请求

## A 口提出的问题
{一个具体、可研究的问题}

## 原始 Owner 意图
{保留用户原话，不用 AI 改写替代}

## 已定义概念
{A 口已经锁定的工作定义}

## 需要查证的事实/概念/案例
- ...

## 不需要查的内容
- ...

## 来源优先级
官方文档 / GitHub repo / issue / README / paper / blog / 社区讨论 / 用户本地文件

## 采用标准
什么证据算支持？什么证据算反驳？什么证据只算参考？

## Context Budget
- Required Context：B 必须读取/压缩的信息
- Optional Context：可参考但不强制的信息
- Excluded Context：不要纳入，避免污染判断的信息
- Retrieve Later：本轮不查，必要时再查的信息
- Durable State：应保存到仓库/文件的信息

## 返回格式
Source Pack + 不确定点 + 对 A 口框架的支持/反驳

## 下一步用途
给 C 口理论生成 / 给 D 口落地 / 给 E 口验证 / 给 F 口判断
```

---

## B 口返回格式

```md
# Source Pack：{主题}

## 1. A 口问题回放
- 原始问题：...
- 本次资料包要回答：...
- 不回答：...

## 2. 关键来源
每个来源要包含：
- 标题/仓库/文件
- URL 或本地路径
- 来源类型：官方文档 / GitHub / paper / blog / 社区 / 本地文件
- 可信度：high / medium / low
- 为什么相关
- 支持、反驳还是修正 A 口框架
- 摘要

## 3. 支持 A 口框架的证据
- 证据：...
- 来源：...
- 对 A 口的影响：...

## 4. 反驳或修正 A 口框架的证据
- 证据：...
- 来源：...
- 需要 A 口修正：...

## 5. 对 C 口有用的材料
- 可直接引用的概念
- 可展开的理论点
- 必须避免的泛化

## 6. 对 D/E/F 的影响
- D 口落地约束：...
- E 口验证关注：...
- F 口需要判断：...

## 7. 不确定点
- ...

## 8. 建议下一步
返回 A 口修订 / 进入 C 口理论 / 进入 D 口落地 / 需要 F 口判断
```

---

## B 口 Blocker Brief

如果 B 口无法完成资料压缩，必须返回：

```md
# B 口 Blocker Brief

## Blocker
卡在哪里。

## Tried
已搜索/读取哪些来源。

## Missing
缺什么关键词、权限、文件、链接或 Owner 判断。

## Risk
继续硬搜会造成什么噪音或误导。

## Suggested Route
建议回 A 口重定义问题，或请求 Owner 补资料。
```

## B 口禁止事项

- 不从模糊话题直接大搜。
- 不把资料原文堆给 C 口。
- 不把来源摘要冒充结论。
- 不替 A 口重新定义目标。
- 不替 C 口生成完整理论。
- 不替 F 口做 Owner 判断。
- 不隐藏反证和不确定点。

---

## 与 A 口的双门关系

```text
A：定义研究问题和采用标准
  ↓
B：先返回 Search Strategy Brief
  ↓
A：批准或修订搜索策略
  ↓
B：执行资料压缩，返回 Source Pack
  ↓
A：判断资料是否进入 C/D/E/F
```

B 口不是自己决定“资料已经够了”。资料够不够，必须由 A 口根据项目目标和下一步用途判断。
