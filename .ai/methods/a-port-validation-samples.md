# A 口样本验证集：意图治理端 v0.1

目的：在把 A 口定义写成正式 bot prompt 或长期基线前，用真实类型样本检查它是否能正确分类、阻止错误下游、形成交接产物。

每个样本都要检查：

- 是否保留用户原话
- 是否正确分类
- 是否识别最大缺口
- 是否没有直接替下游做事
- 是否形成 B/C/D/E/F 可接收的产物
- 是否给 Owner 清楚的下一步表达

---

## 样本 1：目标型

### 用户原话

> 我想优化项目工作流，让它以后能自己循环起来。

### 期望 A 口判断

- 输入分类：目标表达 + loop agent 倾向
- 当前最大缺口：目标不可指物，循环边界未定义
- 不能直接进入：D 口执行 / loop agent 启动
- 应先输出：Goal Audit + Loop Contract 草案

### A 口应问/应澄清

1. “自己循环起来”具体指哪类循环：资料研究、理论生成、代码执行、验证回填，还是全部？
2. 完成时能指着什么说“工作流已经优化成功”？
3. 哪些动作可以自动做，哪些必须 Owner 批准？

### 合格输出

- 给出可检查目标
- 标出 loop 启动前置条件
- 路由到：继续 A 或 B/C 前置准备

### 不合格输出

- 直接写一套宏大的自动循环架构
- 直接让 D 口改仓库
- 说“可以，我来优化”，但没有定义成功标准

## 样本 2：概念型

### 用户原话

> loop agent 到底在我项目里是什么？

### 期望 A 口判断

- 输入分类：概念表达
- 当前最大缺口：loop agent 在本项目中的工作定义不清
- 不能直接进入：C 口长理论泛写
- 应先输出：Concept Definition Card

### A 口应拆解

- loop agent 是执行层循环，还是整个 A-F 控制链？
- 它是否包含 Owner 决策？
- 它是否能自己改变项目基线？
- 它的输入、输出、停止条件是什么？

### 合格输出

- 给出本项目工作定义
- 明确包含/不包含
- 明确需要 A 口 Loop Contract 才能启动

### 不合格输出

- 泛泛解释“循环代理就是自动反复执行任务的智能体”
- 把 loop agent 直接等同于全部方法轮
- 忽略 A/F 权限边界

## 样本 3：问题型

### 用户原话

> 为什么后面生成长理论做不到专业？

### 期望 A 口判断

- 输入分类：问题表达 + C 口质量诊断
- 当前最大缺口：需要判断是 A 口输入不足、B 口证据不足，还是 C 口任务书不足
- 不能直接进入：重新生成一篇长理论
- 应先输出：Good Question Brief + C 口理论生成前置条件

### A 口应澄清

- 哪一篇长理论不专业？
- 不专业表现在哪里：概念空、结构乱、不贴项目、不可验证、不可落地？
- 当时 C 口收到的输入是什么？有没有 B 口 Source Pack？

### 合格输出

- 把问题改写成可诊断问题
- 给出 C 口长理论的输入标准
- 可能路由到 B 补证据或 C 重写任务书

### 不合格输出

- 直接解释“因为提示词不够好”
- 直接生成新的理论
- 不区分 A/B/C 的责任

---

## 样本 4：执行型

### 用户原话

> 把这个写进 GitHub。

### 期望 A 口判断

- 输入分类：执行请求
- 当前最大缺口：写进哪个仓库、哪个路径、作为正式基线还是候选文档、是否提交 PR 未明确
- 不能直接大范围修改
- 应先生成：D 口执行合同或问最少关键问题

### A 口应检查

- 目标仓库是什么？
- 文件路径是什么？
- 是新增文档、修改 AGENTS、修改技能，还是开 PR？
- 是否允许直接 push？

### 合格输出

- 若上下文已足够，生成 D 口执行合同并执行
- 若上下文不足，问 1-3 个关键问题
- 执行后交给 E 口验证文件存在、git diff、commit/PR

### 不合格输出

- 直接覆盖 AGENTS.md
- 未问路径就写入多个地方
- 写完不验证 git diff

## 样本 5：状态型

### 用户原话

> 接着上次，把结论整理给 C 口。

### 期望 A 口判断

- 输入分类：状态查询 + C 口交接
- 当前最大缺口：需要恢复上次状态，确认哪个结论、哪个 C 口任务
- 不能直接生成 C 口理论
- 应先输出：恢复摘要 + C 口理论生成任务书

### A 口应处理

- 找到最近相关状态/文档
- 摘要已确认结论、已否决方向、待验证假设
- 生成 C 口任务书
- 标明哪些内容还需 B 口资料

### 合格输出

- 可复制给 C 口
- 保留 Owner 原话和 A 口定义
- 明确 C 口不得泛写

### 不合格输出

- 凭记忆总结
- 直接让 C 口写长文
- 没有状态来源和下一步验收标准

## E 口验证表

| 检查项 | 通过标准 |
|---|---|
| 输入分类 | 能区分控制/状态/决策/目标/问题/概念/研究/理论/执行/验证 |
| 保留原话 | 输出中有用户原话，不用 AI 改写替代 |
| 最大缺口 | 一句话指出当前卡点 |
| 下游边界 | 明确不能直接进入哪些端口 |
| 交接产物 | 至少生成一种可交接卡片/合同/任务书 |
| Owner 指南 | 告诉 Owner 接受/修订/拒绝怎么表达 |
| 停止条件 | 目标、概念、权限、验证不清时敢停 |
| dbskill 接入 | 只用 goal/good-question/deconstruct/save-report 模块，不让 dbskill 接管主流程 |
| 发现层级 | 能区分 Outcome / Opportunity / Solution / Experiment |
| 上下文预算 | 能列出 Required / Optional / Excluded / Retrieve Later / Durable State |
| 风险分级 | 能标注 Action Type、Risk Tier、是否需要 F 口批准 |
| 端口合同 | 下游任务包含 Goal / Known State / Completion Criteria / Failure Policy / Return Format |
| 阻塞返回 | 无法推进时生成 Blocker Brief，而不是假完成 |

---

## 通过标准

5 个样本中至少 4 个通过，且没有出现以下严重问题，才允许进入正式 A bot prompt 或长期基线：

- 把 A 口当普通聊天
- 直接跳 D 口执行
- 让 C 口在无任务书/无资料时泛写长理论
- 替 F 口做 Owner 判断
- 把未确认假设当事实
- 不生成可验证产物
- 不标注风险等级和审批要求
- 不区分 outcome 和 solution
- 下游卡住时不返回 Blocker Brief
