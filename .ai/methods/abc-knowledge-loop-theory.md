# A/B/C 互相驱动知识循环理论

## 1. 问题陈述与为什么 A-only 会失败

A-only 的常见失败，不是“问得不够多”，而是把三个不同的问题混成一个：

1. 目标到底是什么。
2. 证据到底支持什么。
3. 方案到底长什么样。

如果 A 端同时承担这三件事，它会很快滑向两种坏结果：

- 只做边界拷问，却无法把问题推进到可执行形态。
- 过早给出结论，把缺证据、缺结构、缺决策混在一起处理。

因此，A-only 循环的本质缺陷是：它能判断“要不要继续问”，却不能稳定地区分“缺的是证据，还是缺的是理论结构”。当这个区分缺失时，循环就会无限停留在澄清层，或者反过来把推测当结论。

这个知识循环的目的不是让 A 变强到包办一切，而是让 A 只负责控制面，让 B 负责证据压缩，让 C 负责理论综合。三者互相制衡，才会把模糊输入推进成可落地的工作包。

## 2. A/B/C 角色模型与控制平面边界

### A 端

A 端是需求控制与路由判断端。它回答的问题是：

- 这句话属于什么输入类型。
- 目标、边界、非目标、约束是否清楚。
- 当前最缺的是证据还是结构。
- 是否应该继续循环，还是转端。

A 端不负责把所有材料补齐，也不负责写最终理论。它的核心产物是“下一步应该去哪里”的判断。

### B 端

B 端是证据压缩端。它回答的问题是：

- 读了什么。
- 哪些是可引用证据。
- 哪些是缺口。
- 哪些只是噪音或未验证想法。

B 端不负责最终综合，也不应把源材料直接扩写成方案。它的价值在于把分散来源压成能被 C 使用的结构化输入。

### C 端

C 端是理论综合端。它回答的问题是：

- 这些证据共同支持什么模型。
- 方案或工作流应如何组织。
- 哪些部分可以写进方法文档、计划文档、数据模型或契约。
- 下一步应交给 D、E、F 中的哪一个。

C 端不是执行端，也不是验证端。它的任务是把 A 的边界、B 的证据和工作目标合成为可交付包。

### 控制平面边界

这套模型强调的是逻辑端口，不是多个物理 bot。A/B/C 是同一控制平面里的三个逻辑阶段，状态应该落在仓库文件、issue、spec、plan 或 loop state 中，而不是只留在聊天记录里。

## 3. 知识循环机制与停止条件

### 基本机制

循环的推荐形态是：

```text
A 先路由
→ 缺证据则转 B
→ 证据够但结构不足则转 C
→ C 发现证据不足再回 B
→ A 判断是否继续或转端
```

这里的关键不是固定顺序，而是“按缺口类型转端”。A 负责把缺口分类，B 负责补证据，C 负责补结构。

### 单轮推进原则

每一轮只推进一个最关键缺口。原因很简单：如果一次性问太多，A 会变成调查问卷，B 会变成资料堆，C 会变成泛化作文，最后没有一个端口真正完成它的职责。

### 停止条件

推荐的停止条件有三类：

- 连续两轮没有新增有效信息，停止。
- 2 到 3 轮后只是在修细节而不是改路线，停止并建议转端。
- 同一个证据缺口或同一条结构矛盾连续出现三轮，停止并输出 owner 决策 brief。

停止不是失败，而是防止循环自我复制。知识循环应该在“足够可执行”时收束，而不是在“永远还能再问一点”时继续。

## 4. B 的证据压缩规则

B 的任务不是搜得最多，而是压得最稳。

### 压缩目标

B 输出的不是原始资料堆，而是以下几类信息：

- 关键来源与其可信度。
- 直接相关的事实。
- 与目标无关或弱相关的内容。
- 尚未解决的缺口。
- 哪些地方不能猜。

### 压缩原则

1. 只保留能影响 A 路由或 C 综合的证据。
2. 明确记录“读了什么”和“没读什么”。
3. 不把大段原文搬运进输出。
4. 不把推测伪装成来源。
5. 发现缺口要显式标注，而不是自行补写。

### 压缩格式建议

B 的最佳产物通常是 source pack 或 evidence brief，至少应包含：

- 文件/来源清单。
- 每份来源的作用。
- 共同结论。
- 不一致之处。
- 待补证据列表。

### 特殊边界

B 不能把普通 skill 仓库误判成 B2 成品项目。默认应优先按 skill 层或 pattern-only 处理，只有当仓库实际改变控制平面、工件生命周期或治理方式时，才考虑进入 finished-project absorption 路线。

## 5. C 的理论综合规则

C 的任务不是再读一遍来源，而是把来源变成模型。

### 综合对象

C 要综合的是：

- A 的目标与边界。
- B 的证据压缩结果。
- 当前工作流对 D/E/F 的接口要求。

### 综合产物

C 应产出以下类型的内容之一或多个：

- 理论草案。
- 工作流模型。
- 方案结构。
- 计划骨架。
- 数据或工件契约。
- 运行手册草稿。

### 综合原则

1. 以边界为前提，不重写需求。
2. 以证据为基础，不补空想。
3. 以可交付为目标，不停留在概念层。
4. 以转端为意识，不把理论文档假装成执行完成。

### C 的自检问题

C 写完前应自问：

- 这份理论是否解释了为什么要这样分工。
- 是否明确指出了哪些信息还不够。
- 是否清楚说明了下一步该交给谁。
- 是否保留了回到 B 或 A 的条件。

如果这些问题答不出来，说明 C 还在堆叠材料，没有完成综合。

## 6. 向 D / E / F 的转端规则

### 转 D

当 C 已经形成足够明确的执行包时，转 D。典型信号是：

- 有清晰的目标。
- 有明确的边界与非目标。
- 有可执行的工件或计划。
- 任务已缩小到可落地切片。

D 只应接收边界清楚的包，不应接收“先帮我想明白”的模糊委托。

### 转 E

当需要独立验证、审查或证据检查时，转 E。典型信号是：

- 需要读回路、审查一致性、跑测试或做客观检查。
- 需要确认 C 的理论是否与 B 的证据一致。
- 需要判断是否存在误判、越权或伪完成。

### 转 F

当问题已经不再是知识或实现问题，而是权限、风险或基线决策问题时，转 F。典型信号是：

- 需要 owner 批准。
- 需要决定是否接受基线变化。
- 需要决定是否扩大执行范围。

### 反向转回

如果 D 发现需求仍然模糊，就回 A。  
如果 E 发现证据不足，就回 B。  
如果 C 发现结构不足，就回 B。  

这不是倒退，而是按缺口类型回流。

## 7. 与 Spec Kit 和 B2 finished-project absorption 的关系

这套 A/B/C 知识循环与 Spec Kit 的关系是：它借用了“从需求到规范再到执行”的骨架，但把知识流拆得更细。

Spec Kit 风格的 spine 强调：

```text
constitution / governance → spec → clarification → plan → tasks → verification
```

本理论中的 A/B/C 更早一层，负责把不清楚的输入压成能进入 spec spine 的材料。也就是说，A/B/C 不是替代 spec spine，而是它的上游知识准备层。

finished-project absorption 则是另一条更严格的路。它适用于成熟外部项目，不适用于普通 skill 仓库。关键判断是：

- 这个来源只是提供能力，还是改变控制平面。
- 它是否定义新的工件生命周期。
- 它是否需要 B2 级的完整吸收包。

因此，本循环与 B2 的关系可以概括为：

- B2 是成熟项目吸收的方法。
- A/B/C 是吸收前的知识生成和综合机制。
- 不是所有 source pack 都应升级为 B2。

## 8. 与 skill repositories 和 intake policy 的关系

skill repository intake policy 解决的是“外部仓库到底该进 skill 层，还是进 B2”。

这条政策与 A/B/C 循环的关系是分层的：

- A 决定问题类型和路线。
- B 判断外部资料是什么类型。
- C 把判断结果转成可用方法。
- intake policy 决定这个来源该被视为 skill layer、pattern only，还是 B2。

这里的核心防线是：不要因为一个仓库“看起来有用”就把它升格为基础方法。多数 skill 仓库应留在 skill_layer 或 pattern_only，只有少数真正改变控制平面者才进入 B2。

这也意味着，A/B/C 的输出在碰到外部仓库时，必须显式给出“是否需要 B2”的判断依据，而不是默认吸收。

## 9. 失败模式与反模式

### 失败模式

1. A 变成万能主持人。
2. B 变成资料仓库，不做压缩。
3. C 变成作文机，不做结构化。
4. 循环一直问，不转端。
5. 证据不足却提前综合。
6. 只改 prompt，不改工件和决策记录。

### 反模式

- 把“多轮澄清”误当作“已经理解”。
- 把“读了很多”误当作“证据充足”。
- 把“写得完整”误当作“已经可执行”。
- 把 skill 仓库误当成成品项目。
- 把工作流说明本身当成目标，而不是把目标压实到可执行包。

### 纠偏信号

如果连续两轮没有有效新增信息，或者反复围绕同一个细节打转，说明应该收束而不是继续发问。  
如果输出里缺少明确转端建议，说明角色边界已经开始漂移。  
如果 B 和 C 的内容无法被 D/E 消费，说明综合层失败了。

## 10. 推荐的提示词 / 运行时结构

推荐的运行时结构不是单一 prompt，而是一个带状态的控制回路：

```text
trigger
→ current understanding
→ primary route inferred
→ A / B / C role block
→ confirmed
→ still missing
→ next best question
→ next action
→ stop or continue
```

### 运行时约束

1. 先输出路由判断，不要先让用户选 A/B/C。
2. 每轮只问一个最关键问题。
3. A 端只负责控制，不负责替 B 或 C 做事。
4. B 端只给证据压缩，不给最终方案。
5. C 端只做理论综合，不冒充执行或验证。
6. 连续无增量时必须停止。
7. 一旦达到执行、验证或 owner 决策门槛，及时转端。

### 状态记录建议

状态应落到可追踪工件中，例如：

- source pack。
- spec / plan。
- loop run state。
- owner brief。
- review notes。

不要把聊天记录当作唯一真相源。

## 11. 建议后续工件

如果这份理论要继续推进，建议补齐这些工件：

1. A/B/C 端口契约的正式版。
2. A/B 双闸门循环的专门说明。
3. C 端理论综合的模板。
4. D/E/F 的转端判定表。
5. 典型失败回放与回收规则。
6. 面向 skill repository 的 intake 决策树。
7. 面向 Spec Kit 的桥接说明。

这些工件会把本理论从“解释模型”推进到“可执行方法”。
